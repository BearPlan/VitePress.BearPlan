- 与用户沟通、最终回复、文档说明、代码注释默认使用中文；必要时可保留英文专业名词，并在首次出现时补充简短中文说明。
- 生成或修改代码时，应为关键业务逻辑、复杂流程、非显然取舍添加简明中文注释；不要为一眼能看懂的赋值、导入、结构体字段添加空泛注释。

## 计划与任务

> 完整方法论（分阶段流水线、子 Agent 状态协议、并行判定、验证铁律）见 Obsidian [[Agent 工作流/index|Agent 工作流专题]]，本节只列执行要点。

- 每次新对话或新任务默认先计划再执行；复杂任务必须给出分阶段计划，简单任务用一两句话说明意图即可。
- 计划阶段先理解目标、影响范围、相关文件和验证方式；涉及项目结构或长期约定时同步检查 Obsidian 索引。
- 计划应简洁可执行，执行中发现不适用时应先说明调整原因再继续推进。
- 用 `TodoWrite` 跟踪多步任务：进入实现前一次性建好全部 todo，同一时刻只一个 `in_progress`，该项验证通过后才标 `completed`。
- 任务粒度：一步相当于 2-5 分钟、一个可独立验证的动作；禁止 TBD / TODO / "稍后实现" 等占位符。

## 子 Agent 调度

- 复杂实现或多文件调查可派子 Agent；两三行的小修留在主会话。
- **上下文隔离**：子 Agent 不继承本会话上下文与历史，由主会话精确构造其完成任务所需的全部信息（Task 全文、场景背景、依赖、提问许可），不让子 Agent 自行翻计划文件。
- **并行判定**：互相独立、无共享状态的调查可并行派多个子 Agent；连续的实现步骤一律串行（共享文件状态，并行会冲突）；需要理解全局状态的任务留在主会话。
- 子 Agent 按四态报告（`DONE` / `DONE_WITH_CONCERNS` / `BLOCKED` / `NEEDS_CONTEXT`），主会话据此决定是否进入审查或补充上下文重派；绝不允许忽略 escalate 或同模型无变化重试 BLOCKED 任务。
- 实现类 Task 完成后过两道审查，顺序固定：先 spec 合规（是否跑题、多建少建），再代码质量（命名、可维护性、是否符合书写规范）。自审不能替代正式审查。

## 开发原则

- 只改必需的部分，优先复用现有成熟代码，避免重复造轮子；保持简单直观，不为假想需求过度设计。
- 架构设计时让边界情况自然融入常规逻辑，而不是单独打补丁。
- 代码应表达真实业务逻辑，删除不再使用的代码、混淆项和无维护价值的兼容层。
- 单个文件不超过 1000 行；接近或超过时按职责拆分模块或工具函数。
- 项目特定的技术栈、目录权限、框架约束等规则，查阅 Obsidian 项目文档。

## 测试与验证

- 所有变更必须通过对应的 lint/format/test 检查，禁止依赖后续修复来兜底。
- 具体的检查命令和验证步骤，查阅 Obsidian 项目文档。
- 新增或变更功能时，应优先补充现有体系下的单元测试或集成测试；如项目没有对应测试基础，不因单个任务强行引入新测试框架。
- **验证铁律**：任何「完成」声明必须有本次消息内新鲜的验证证据（测试输出、编译结果、lint 通过），"之前跑过"、"应该没问题"不算；不盲信子 Agent 的 success 自述，对其委派结果用 diff 或日志独立确认。
- 若因环境、依赖或外部服务限制无法完成某项检查，最终回复必须明确说明未执行的检查和原因。

## 文档维护

- `AGENTS.md` / `CLAUDE.md` 只记录稳定的工作规则、质量要求和项目入口；Obsidian 记录长期知识（架构、API、模块职责、技术栈、踩坑记录、跨项目可复用经验）。
- **通用库持续精进**：通用知识库（`net core` / `vue3` / `Agent 工作流`）应跟随项目开发持续进化；项目沉淀的跨项目通用经验（框架模式、语言特性、工具配置、可复用架构套路）必须反哺到对应技术栈通用库（剥离项目命名与业务细节），项目文档只留项目特定部分与指向通用文档的链接。不得让通用库长期滞后于项目实际。
- 模块边界、入口文件、API 端点、配置结构或公共约定发生变化时，必须更新对应 Obsidian 文档。
- 只有长期有效的模式、约束或关键决策才沉淀为 Obsidian 专题文档；一次性过程记录和完成汇报不写文档。确需新增时在项目索引中补充条目。
- Obsidian 文档必须同步维护索引和互链：新增/移动/改名时更新相关链接，删除/改名前先搜索旧引用避免断链。跨项目通用知识沉淀到 Vault 通用目录。

## 任务处理

- 需求不明确时，先结合代码和文档自行确认可发现事实；仍存在高影响歧义时，再向用户提问。
- 对用户设想应结合项目现状客观评估；发现更稳妥方案时，说明理由并给出可执行替代方案。
- 复杂任务应先抽象出目标、边界、影响面和验收标准，再拆分步骤推进。
- 遇到阻塞立即停下上报，不猜、不硬冲；连续多次修改仍无法解决问题时，应停止叠加补丁，回到更早状态重新分析根因并换方案。

### 任务回顾

在任务完成、呈现最终消息前，必须执行以下回顾：

- 根据任务执行情况判断是否需要更新 Obsidian 文档：若产出新的可复用模式、跨模块约束或关键决策，则写入对应文档；若仅影响局部实现且无长期价值，则无需写回。
- 如 Obsidian 索引中记录了当前任务文档，检查其 TODO 完成情况；已全部完成则将其移动至 `workflow/done/` 并从索引的"当前任务文档"部分移除。
