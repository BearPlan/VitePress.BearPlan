## 写作风格约束（来自现有文档惯例）

- H1 标题 + blockquote 引言（一句话定位）
- 章节顺序：定位 → 关联文档导览 → 正文（H2 分组）→ 关联资源
- 容器语法：`::: tip 标题` / `::: warning 标题`（带短标题，不用裸 `::: tip`）
- 代码块带语言标签，命令前用 `#` 注释意图
- 中文为主，术语保留英文（DTO、AOP、swagger、OpenAPI）
- 收尾「关联资源」统一列：主项目仓库 / Core 仓库 / 开发文档 bear.js.org / 作者微信
- 锚点显式声明（`### 标题 {#id}`）

## alovajs.md 重写大纲（项目落地导向）

```
# Alova.js
> 一句话定位：BearPlan.Admin 的请求层，hook 自管 loading/data，配合 Worma 自动生成的 API 代理使用。

## 这是什么
- 与 axios 的差异（hook 化、状态自管）
- BearPlan 为何选 alova（与 Worma 自动生成配套）

## 关联文档
- 内联链接到 [Worma](/core/worma)（自动生成的 API 代理从这里来）
- 整体架构见 [框架介绍](/core/framework)

## 在 BearPlan 中的位置
- 一张小流程图：后端 swagger → Worma 生成 Apis → 页面 useRequest 调用
- 前后端协同卖点的一句话回顾（不重复 framework.md 细节）

## 实例与全局拦截器（createAlova）
- baseURL / beforeRequest 注入 Bearer
- responded.onSuccess：自动解包 { data: T }、204 放行、双格式响应（统一格式 vs 原始响应）放行
- ::: tip 关键约定：页面不写 try/catch，异常交底层拦截器

## 核心 Hooks 速览
- 表格：场景 → hook（useRequest / useForm / usePagination / useSSE）→ 一句话用途
- 每个 hook 给一个最小示例（3-5 行，BearPlan 风格）

## 命名规范（项目约定）
- send 别名 camelCase + 动词前缀（getDevices / deleteDevice）
- loading 别名加 Loading 后缀
- 三参数必写：immediate / initialData / force
- 箭头函数单参数省括号：transform: res => ...
- ::: warning 这些是 BearPlan 项目约定，详见书写规范

## 关联资源
- alova 官方文档 https://alova.js.org/
- 主项目仓库 / Core 仓库 / bear.js.org / 作者微信
```

**篇幅目标**：与 `quickstart.md`（~110 行）相当，约 150-180 行。
